From the `fable_mockup_v4.html` review (2026-06-18), the following gaps need resolution before or during the phases noted:

**D-034 opened (Phase 17):** Openings model — `MaintainedTruth` needs optional `group` and `effect_text` fields, or a separate `Opening` entity. FABLE Ledger categories (Position / Opening / Leverage) must be preserved in backend data.

**Phase 21 — CASEFILE / Known Facts:** `window.CASEFILE` is currently a static JS object. Needs to be a live projection of `committed_facts()` filtered to revealed facts in the player's POV. The CASEFILE groups (People / Places / Known Truths / Clues / Open Questions / Promises & Debts) are presentation categories that layer on top of `epistemic_type` (see D-032). The channel router or a dedicated projection step must annotate facts with a CASEFILE category.

**Phase 21 — "Just Changed" feed:** `recent` events in the frontend are plain text strings. The real system needs human-readable summaries of `effect_applied` events, generated by the narrator or a dedicated render step. Backend must define the `recent` event payload format and who generates it.

**Phase 21 — Hidden NPC rail filter:** NPCs with `presence: "hidden"` must not appear in the character rail — knowing the avatar exists is already information. Frontend rail rendering must gate on `presence` state.

**Phase 15+ — Session config schema:** `SETTINGS_JSON.pacing.seam_threshold` and `action_budget` are adjudicator configuration with no backend counterpart yet. Define a session config object the adjudicator reads. Consider adding to Phase 13 scope or opening a decision.

**Why:** Per fable_mockup_v4.html assessment — these gaps don't block Phases 13–16 but will create retroactive schema conflicts if left undocumented until Phase 21.
